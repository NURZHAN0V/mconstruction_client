# Строгие правила разработки

## Типизация
- Запрещено использование типа any. Всегда использовать конкретные типы.

## Структура проекта
- Строго придерживаться структуры папок Nuxt версии 4.
- Правильно организовывать папки и компоненты в директории components.

## Зависимости
- Запрещено устанавливать новые зависимости без явного разрешения пользователя.

## Переменные окружения
- Переменные окружения не должны попадать на клиент. Секреты (токены, ключи API, идентификаторы) хранить только в серверном runtimeConfig (без public). Обращение к внешним API выполнять только с сервера (server/api, server utilities).

## Пользовательский интерфейс
- Интерфейс должен быть адаптивным и доступным для всех пользователей и устройств.
- Всегда использовать скелетон при ожидании данных от сервера или внешних сервисов.

## Анимации
- Создавать анимации только средствами CSS (transition, animation, @keyframes).

## Иконки
- Запрещено использовать SVG иконки. Использовать только иконки RemixIcon.

## Цветовая палитра
- Использовать в приоритете цвета из палитры, заданной в tailwind.config.js.
- Запрещено использовать произвольные hex-значения и цвета вне палитры.

## Документация
- Строго следовать документации, расположенной в каталоге docs.

## Рендеринг
- Не использовать серверный рендеринг. Приложение работает только на клиенте.

## Хранение данных
- Использовать куки для хранения данных на стороне клиента.
- Использовать куки для работы с пользовательскими данными, настройками и сессиями.

## Импорты
- Использовать автоматический импорт Nuxt для всех установленных модулей. Явные импорты не требуются.

## Управление состоянием
- Использовать Pinia для управления состоянием приложения.

## Качество кода
- Следовать лучшим практикам разработки.
- Писать минимальный необходимый код.
- Избегать сложных функций. Код должен быть простым и понятным.
- Всегда комментируй участки кода на русском языке.

## Отладка
- Выводить отладочные логи для выявления проблем в процессе разработки.
- Обязательно удалять все отладочные логи после решения проблемы и перед коммитом изменений.
- Оставлять только необходимые логи для обработки ошибок
